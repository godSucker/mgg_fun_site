# Анализ логики скрещивания сайта и roadmap внедрения шансов/времени

Документ подготовлен на основе:
- реверс-инжиниринга декомпилированной функции `getPossibleOutputSpecimens` (ARM64/Ghidra) игрового клиента;
- реального `breedingsettings_decoded.xml` (`SpecimenHybridization`, `Stars`);
- `mutants.json` (поля `genes`, `type`, `chance`, `incub_time`);
- десятков реальных пар шансов/времени, собранных вручную и сверенных с формулами до сотых процента / до секунды;
- разбора 5 файлов текущей кодовой базы сайта (`core-algorithm.ts`, `type-filters.ts`, `breeding.ts`, `engine.ts`, `logic.ts`).

Все формулы ниже **эмпирически подтверждены** — не гипотезы, а формулы, воспроизводящие реальные проценты/время игры без единого расхождения на всех проверенных примерах (одноген×одноген, одноген×разноген, разноген×разноген, обычные/легенды/герои/PVP/секреты, все 3 уровня скрещивалки, все 4 состояния звезды включая платину).

## 1. Что в коде уже сделано верно

### 1.1 `core-algorithm.ts` — генерация возможных генных комбинаций
`calculatePossibleOffspring()` реализует ровно то же правило, что и в декомпилированном коде игры:
- Если гены родителей совпадают (`g1 === g2`) → всегда возможен моноген (`AA`), а одноген (`A`) — **только если хотя бы один из родителей сам одноген** (`!bothParentsTwoGene`). Это соответствует правилу «одноген может выпасть, только если его ген есть у обоих родителей и хотя бы один родитель сам одноген» — то есть 2ген×2ген никогда не даёт одноген на выходе.
- Если гены разные → добавляются оба порядка (`AB`, `BA`).

Это верно и подтверждено на всех проверенных примерах. **Менять не нужно.**

### 1.2 `type-filters.ts` — фильтрация по типу мутанта
Списки `UNBREEDABLE_TYPES`, `INHERITANCE_ONLY_TYPES` и `BREEDABLE_LEGENDS` **полностью совпадают** с тем, что мы независимо вывели эмпирически из реальных пар шансов:
- `UNBREEDABLE_TYPES` (secret, recipe, videogame, zodiac, event, special, gacha, community, seasonal, boss) — никогда не участвуют в обычном скрещивании. Подтверждено.
- `INHERITANCE_ONLY_TYPES` (heroic, hero, pvp) — участвуют только если сам этот мутант — один из родителей. Подтверждено (пример: A_01×AA_01, где ни один HEROIC/PVP-не-родитель не выпал).
- `BREEDABLE_LEGENDS` — легенды-исключения, которые ведут себя как обычные (не требуют быть родителем). Реальный размер белого списка — 26 (совпадает с количеством уникальных `RU`+`EN` записей за вычетом дублей на секреты).

**Единственное, что стоит перепроверить:** в `BREEDABLE_LEGENDS` есть `'hor'` и `'gore'` (английские) без соответствующих кириллических пар — а в `secretCombos.ts` (`Гор` как ребёнок секретных рецептов `Ктулху+Ксенос` и `Лорд Преисподней+Благородный дракон`) уже есть легенда с похожим именем как **СЕКРЕТ**, не как «breedable legend». Так как имена мутантов в `mutants.json` — кириллические (`m.name`), запись `'hor'`/`'gore'` без кириллического эквивалента, скорее всего, никогда не сработает (мёртвый код) — если это не два *разных* мутанта, стоит сверить и либо добавить кириллический вариант, либо убрать как лишнее.

## 2. Найденные проблемы

### 2.1 ⚠️ КРИТИЧНО: `engine.ts` использует неверное сопоставление `LengthOut`

В `engine.ts` есть закомментированная (предположительно неиспользуемая, но нужно проверить, не подключена ли она где-то ещё) попытка расчёта весов:

```ts
// 'one' => homogeneous pairs AA, BB
// 'two' => hetero pairs AB, AC
// 'three' => SINGLE-LETTER children (A, B, C, ...)
```

**Это неверно.** Реальное сопоставление (подтверждено decompile + XML + десятками реальных примеров):

| Поле XML | Что на самом деле означает |
|---|---|
| `LengthOut.one` | вес для потомка **длины 1** (одноген: `A`, `B`, …) |
| `LengthOut.two` | вес для потомка **длины 2** (моноген `AA` **и** разноген `AB` — оба варианта, без разделения!) |
| `LengthOut.three` | вес для потомка длины 3 (в игре таких мутантов нет, поле мёртвое — подтверждено `BREEDING_SUMMARY.md`) |

То есть деление не «моноген / разноген / одноген», а **по количеству генов у потомка, точка**. `AA` и `AB` — это ОДНА и та же категория `two`, они делят её вес пропорционально своим `odds`, а не делятся на разные XML-поля.

**Доказательство от противного:** возьмём пример A_01×A_05 (оба родителя — один и тот же ген `A`, категория `allele1=1,allele2=1`, `LengthOut = {one:1, two:4, three:0}`). Реальные данные игры показывают, что одногенные потомки (`A_01`, `A_05`, `A_13`, `A_14`) выпадают с ненулевым весом (используется `LengthOut.one=1`). Если использовать логику `engine.ts` (одногенный потомок = поле `three`), их вес был бы `×0`, то есть они были бы **невозможны** — что прямо противоречит реальным данным игры. Баг подтверждён математически, не как гипотеза.

**Рекомендация:** либо полностью выпилить `engine.ts` как неактуальный черновик, либо переписать `predict()` по формуле из раздела 3.

### 2.2 Полностью отсутствует система «мягких» рецептов (soft recipes) для типа `RECIPE`

`secretCombos.ts` (жёсткие рецепты, 22 шт.) даёт гарантированный 100%-й результат при точном совпадении двух конкретных родителей — это подтверждено и реализовано верно.

Но в игре есть **второй, отдельный механизм** для мутантов с `type: RECIPE` (Секреты): они могут получиться **вероятностно**, наравне с обычными кандидатами, но только если пара родителей соответствует **отдельному, персональному «мягкому» рецепту именно у этого мутанта** (аналог поля `getRecipes` в декомпилированной функции игры). Пример: `BE_04` (тип `RECIPE`, `chance=120`) выпадает с вероятностью наравне с обычными кандидатами **только** при скрещивании конкретно `CB_01 × CE_01`, но нигде больше.

Сейчас в `type-filters.ts` **весь** тип `recipe` жёстко в `UNBREEDABLE_TYPES` → значит, такие мутанты **никогда не появятся** в расчёте сайта, даже когда родители совпадают с их персональным рецептом. Это реальный пробел функциональности (не баг вычисления, а отсутствующая фича) — сайт будет показывать неполный список исходов для таких пар.

**Что нужно:** отдельная таблица `specimenRecipes: Record<mutantId, Array<{allele1Genes: string, allele2Genes: string}>>` (или на уровне пары родителей), и в `canBreedByType`/`calculateBreeding` — доп. ветка: если тип `RECIPE` и пара родителей совпадает с персональным рецептом этого мутанта → он допускается в общий вероятностный пул (не как гарантированный секрет, а именно как весовой кандидат — см. формулу в 3.3).

**Как получить эти данные:** такой список рецептов для RECIPE-мутантов, скорее всего, зашит в те же игровые ресурсы, что и `SpecimenHybridization` (десериализуется тем же `serializeToXmlNode`, судя по декомпилу — `SpecimenDefinitionHelper::getRecipes`). Нужно поискать в ассетах игры XML/данные с рецептами по каждому `RECIPE`-мутанту.

### 2.3 Полностью отсутствует расчёт шансов (odds/probability)

`calculateBreeding()` в `breeding.ts` находит **список** возможных потомков и сортирует их по `odds` по убыванию — но **нигде не вычисляется вес и процент**. `LengthOut` из XML вообще нигде не используется в текущей "рабочей" ветке (`core-algorithm.ts` + `breeding.ts`), только в неиспользуемом `engine.ts` — и там неверно (см. 2.1).

Это главное, что нужно добавить — см. раздел 3.

### 2.4 Полностью отсутствует расчёт времени скрещивания

Нигде в 5 файлах нет формулы длительности. `mutants.json` уже содержит нужное поле `incub_time`, этого достаточно — формула в разделе 4.

## 3. Полная формула шансов (валидирована на всех типах пар и всех уровнях)

```
эфф_odds(кандидат) = odds(кандидат)
                    + levelBonusOdds(уровень_скрещивалки)
                    + [тип(кандидат) == RECIPE] × levelBonusOddsReceipe(уровень_скрещивалки)

вес(кандидат) = LengthOut[длина_кандидата] × эфф_odds(кандидат)

шанс(кандидат) = вес(кандидат) / Σ(вес всех допущенных кандидатов) × 100%
```

### 3.1 Таблица `LengthOut` (по длине ДНК каждого из двух родителей, из XML `SpecimenHybridization`)

| allele1 (длина р.1) | allele2 (длина р.2) | one (вес потомка-1ген) | two (вес потомка-2ген) | three |
|---|---|---|---|---|
| 1 | 1 | 1 | 4 | 0 |
| 1 | 2 | 1 | 18 | 1 (мёртвое поле) |
| 1 | 3 | 1 | 15 | 5 |
| 2 | 2 | 0 | 1 | 1 |
| 2 | 3 | 0 | 1 | 3 |
| 3 | 3 | 0 | 0 | 1 |

(Пара `allele1`/`allele2` не упорядочена — искать нужно в обе стороны, `(1,2)` и `(2,1)` — это одна и та же запись.)

Триген в игре не существует (`incub`/`type` для таких отсутствуют) — `LengthOut.three` можно игнорировать/не имплементировать, поле мёртвое.

### 3.2 Таблица бонуса уровня скрещивалки

| Уровень | `levelBonusOdds` |
|---|---|
| 1 (база) | 0 |
| 2 | 50 |
| 3 | 200 |

Это буквально значения `level2BonusOdds`/`level3BonusOdds` из `<Hybridization>` в XML — сложность лишь в том, что этот бонус **не множитель**, а **флэт-число, прибавляемое напрямую к `odds` каждого допущенного кандидата** (не к финальному весу и не к `LengthOut`).

### 3.3 Отдельный доп. бонус для RECIPE-типа

| Уровень | `levelBonusOddsReceipe` |
|---|---|
| 1 | 0 |
| 2 | 1 |
| 3 | 4 |

Прибавляется **дополнительно** к обычному `levelBonusOdds`, но **только** к тем кандидатам, у которых `type === RECIPE` и которые допущены в пул через персональный «мягкий» рецепт (см. 2.2). У обычных/легенд/героев/PVP этой добавки нет.

### 3.4 Правила допуска кандидата в пул (порядок важен)

```
1. Если пара родителей точно совпадает с одним из 22 жёстких secretCombos →
   ответ детерминирован (100% этот мутант), дальнейший расчёт не нужен.

2. Иначе: пул_генов = гены(родитель1) ∪ гены(родитель2)
   Для каждого мутанта m из базы:
     a. genes(m) ⊆ пул_генов?  — если нет, пропуск
        (технически точнее — генерировать через calculatePossibleOffspring
         как сейчас и делать сравнение кодов, это уже верно реализовано)
     b. LengthOut[длина(m)] > 0 для данной пары allele1/allele2? — если нет, пропуск
     c. по типу:
        - в UNBREEDABLE_TYPES (кроме recipe, который обрабатывается отдельно ниже) → пропуск
        - type === RECIPE → допускается ТОЛЬКО если пара родителей входит
          в персональный мягкий рецепт этого мутанта (раздел 2.2); тогда
          используется бонус 3.3
        - type в {heroic, hero, pvp} → допускается, только если m — один из родителей
        - type === legend/legendary → допускается, если m — один из родителей
          ИЛИ m в белом списке BREEDABLE_LEGENDS
        - default → допускается всегда
3. Для всех допущенных — посчитать вес по формуле из начала раздела 3,
   сумму весов, проценты.
```

## 4. Полная формула времени скрещивания (валидирована на всех уровнях и звёздах, включая платину)

```
duration_сек = (incub_time(родитель1) + incub_time(родитель2)) [минуты] × 60
             × speed(уровень_скрещивалки)
             × starMultiplier(звезда)
```

### 4.1 Скорость по уровню скрещивалки

| Уровень | `speed` |
|---|---|
| 1 (база) | 0.99 |
| 2 | 0.66 |
| 3 | 0.33 |

(Чем выше уровень здания, тем меньше множитель — то есть быстрее. Ровно 1/3, 2/3 и почти полная величина.)

### 4.2 Множитель звезды (`timerMalus` из `<Stars>`, переведён в множитель по формуле `1 + timerMalus/100`)

| Звезда | `timerMalus` (XML) | Множитель |
|---|---|---|
| нет | — | 1.0 |
| Bronze | 50 | 1.5 |
| Silver | 100 | 2.0 |
| Gold | 200 | 3.0 |
| **Platinum** | 0 | **1.0** (совпадает с «без звезды»!) |

### 4.3 Особый случай — Platinum / `useFusionRules`

Подтверждено пользователем и цифрами: платиновая звезда требует **двух одинаковых мутантов, оба с золотой звездой** (два разных золотых мутанта — нельзя). Время считается по **той же самой формуле** (раздел 4, множитель звезды = 1.0, как «без звезды»), но **исход** — не через обычный вероятностный алгоритм генов/типов, а полностью отдельная механика **гарантированного результата: 2 копии того же самого мутанта**, без участия `LengthOut`/`odds`/белых списков вообще. Это и есть `useFusionRules="true"` из XML — отдельная ветка логики, включающаяся именно при self×self на голде→платине.

**Нужно доп. проверить (нет данных):** что происходит на платине при скрещивании двух **разных** мутантов (если такое вообще возможно по условиям игры — по словам пользователя, нет, нужны именно 2 одинаковых золотых). Если platina действительно требует строго одинаковых родителей — эта ветка вообще не пересекается с обычным вероятностным алгоритмом, и реализовать её можно как простой отдельный if: `if (p1.id === p2.id && p1.star === GOLD && p2.star === GOLD) return { guaranteed: platinumVersionOf(p1), count: 2 }`.

## 6. Что подтверждено эмпирически, а что — предположение

**Подтверждено многократно, на разных типах пар, разных уровнях, разных звёздах (точное совпадение процентов/секунд с реальными данными игры):**
- Формула весов (раздел 3), включая level-бонус и отдельный recipe-бонус;
- Таблица LengthOut для всех 6 категорий длин;
- Правила допуска по типу (раздел 3.4), включая белый список легенд;
- Формула времени (раздел 4.1–4.2), включая точное значение констант скорости.

**Предположение / нужна доп. проверка (мало данных):**
- Платина при разных родителях (не self×self) — не проверялось, возможно, невозможно в принципе;
- Существует ли похожий self×self-fusion на Bronze/Silver/Gold звёздах (не только на платине) — не проверялось;
- Точный список персональных «мягких» рецептов для всех RECIPE-мутантов — есть только один пример (BE_04), полного списка нет;
- Полный список 26 breedable-легенд не сверялся построчно с реальными игровыми данными (только по количеству элементов множества, раздел 1.2).
